# Analytics Engineer — Technical Interview Challenge

## Context

This repo implements a dbt-based analytics stack with:

- raw projection models
- Data Vault 2.0 models (hubs, links, satellites)
- consumer-facing dimensional/fact models
- local DuckDB profile and Snowflake profile examples

The code was generated by AI and contains intentional issues. Some are architecture issues, some are runtime/performance/integrity problems, and some are subtle implementation details.

**Goal:** Validate if the repository is production-ready for a Snowflake + dbt team that uses AI-native development.

## Interview Tasks

### Task 1 — Deep Code Review

You must produce a list of **exactly 10 issues** grouped by severity:

- **High (architecture / correctness)**
- **Medium (performance / integrity / maintainability)**
- **Low (implementation details / naming / consistency)**

For each issue, provide:

1. **Where** — file + section
2. **What** — precise issue description
3. **Why it matters** — production impact
4. **How to fix** — corrected approach (SQL/config level)

### Task 2 — Snowflake-Specific Validation

Review `profiles.snowflake.interview.yml` and identify Snowflake-specific risks, anti-patterns, or misconfigurations.

### Task 3 — AI Workflow Validation

Evaluate how safe this repository is for AI-native development:

1. Which failures could slip through if teams trust generated code too quickly?
2. Which validation gates should be mandatory (human review + automated checks)?
3. Which parts of `AGENTS.md` help, and which parts are incomplete or unsafe?

## What We Assess

- Data Vault 2.0 fundamentals (hub/link/satellite semantics, hash key/hashdiff correctness, SCD2 behavior)
- dbt modeling conventions and dependency design
- Snowflake practical knowledge (roles, warehouse strategy, schema/database behavior, cost/performance implications)
- Ability to validate AI output against source mappings and engineering standards
- Ability to communicate risk in a production review
